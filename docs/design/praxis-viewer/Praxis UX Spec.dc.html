A viewer for the artifact trail

A static web UI where a user enters or uploads a document, runs the Praxis pipeline once, and steps through each pass with its artifacts. The UI inspects the completed trail — it adds no transformation logic of its own.

Surface
One page · two zones
Engine
Python via Pyodide
Deploy
Static · no backend
Passes
6 · one panel each
01 — Goals & non-goals

Goals

  • → Enter a document by paste, upload (.md / .txt), or a bundled example.
  • → Step through the six passes, one screen per pass.
  • → Trace the evidence chain: transformation → recommendation → observation.
  • → Download the complete artifact trail — the same six files the CLI writes.
  • → Deploy as a static site on GitHub Pages or Vercel with no backend.

Non-goals (v1)

  • × No accounts, persistence, or server storage — documents never leave the browser.
  • × No editing of transformation packs (view-only).
  • × No accept/reject of individual transformations mid-run.
  • × No LLM calls — the pipeline stays deterministic and auditable.
02 — The one page

The pipeline executes in milliseconds, so the UI does not animate execution. It runs the whole pipeline once, then lets the user walk the completed artifact trail. Stepping is inspection, not execution — consistent with "the artifact trail is the product."

Anatomy · one page, two zones
Zone A — persistent pass stepper
1 Input → 2 Observe → 3 Recommend → 4 Transform → 5 Validate → 6 Report
Zone B — pass panel (content of the active step)

Run lifecycle & stepper states

Before a run

Only Input is enabled. Steps 2–6 are dimmed and non-interactive.

After a run

All steps enabled, each with a summary badge. Free navigation in any order.

On edit

Editing the input invalidates the run — steps 2–6 grey out until re-run.

04 — Safety color code

Used consistently across highlights, badges, and diffs. Colors must meet WCAG AA contrast and are always paired with the text label — never color alone.

Note: the RFC's green / amber / purple maps onto DHK's semantic accents — teal for safe, orange for low-risk, purple for review — since the Electric Cobalt palette carries no green or amber.

05 — Architecture

Run the Python harness in the browser

The praxis package is stdlib-only and small. Run it unchanged in the browser via Pyodide (CPython compiled to WebAssembly) so the UI stays evidence of what the harness actually does.

Why not a TS port?

A port is a second implementation of the rules. The two drift, and the UI stops being evidence. One source of truth outweighs load time.

Why not serverless?

It rules out GitHub Pages, adds a network hop and a privacy question, and still packages the same Python. Client-side keeps both targets open.

Cost: a one-time ~7 MB Pyodide download. Mitigations — load in the background while the user types, cache via service worker, show "loading engine" status until ready. The engine boundary below keeps it swappable.

Engine boundary

The UI calls a single function and receives the full artifact trail. TypeScript types mirror the dataclasses in praxis/models.py exactly. Only this module knows Python exists.

interface PraxisResult {
  observations:    Observation[];      // observations.json
  recommendations: Recommendation[];   // recommendations.json
  transformations: Transformation[];   // transformations.json
  validation:      Validation;         // validation.json
  final:           string;             // final.md
  report:          string;             // report.md
  metrics: { before: Metrics; after: Metrics };
}

runPipeline(source: string): Promise<PraxisResult>

Harness prerequisite — the only change to the existing package: extract the pipeline core from cli.run into praxis.pipeline.run_pipeline(text: str) → dict that returns the artifacts instead of writing files. The CLI and the UI shim both call it.

06 — Acceptance checks
01Pasting the golden example and running produces artifacts byte-identical (JSON/Markdown content) to python -m praxis run on the same input.
02Every transformation in step 4 links to a recommendation in step 3 and an observation in step 2.
03A document with no matches runs cleanly: empty observation list, no diff, validation pass.
04A 50 kB document runs without freezing the UI — the pipeline executes in a web worker.
05The site works offline after first load, via service worker.
07 — Future · out of scope for v1
Accept / reject

Approving individual recommendations before transform — turns the viewer into a review tool.

Pack authoring

Pack selection and editing. v1 is view-only, single pack.

Run permalinks

Shareable links — needs URL-encoded documents or storage; revisit privacy first.
